\section{Conclusiones}

Las conclusiones del proyecto se apoyan en una distincion central: el prototipo funcional existe y su arquitectura muestra viabilidad funcional preliminar y coherencia arquitectonica, pero la validacion empirica final depende de la ejecucion posterior de pruebas de usabilidad y rendimiento sobre la build congelada.

\subsection{Respecto al objetivo general}

Se logro construir un visor web interactivo para la visualizacion tecnica del Holybro X500~V2, accesible desde navegador y capaz de integrar navegacion 3D, seleccion de piezas, informacion contextual y herramientas de analisis visual. Esta base sostiene una viabilidad funcional preliminar del enfoque para el programa de Ingenieria Multimedia y para escenarios de visualizacion tecnica en la web. No obstante, la viabilidad tecnica final en terminos de KPIs y validacion empirica debe cerrarse con la build congelada y las mediciones pendientes.

\subsection{Respecto a los objetivos especificos}

\paragraph{OE1: pipeline de optimizacion y preparacion de activos.} Se consolidaron el caso de estudio, la taxonomia canonica y un pipeline defendible que combina importacion CAD, remodelado hard-surface, modularizacion de fasteners, automatizacion puntual y preparacion para runtime web. El FBX runtime final y el importador Unity quedaron preparados; no obstante, la verificacion post-import, la aplicacion de materiales/texturas finales y la medicion posterior siguen siendo pendientes tecnicos del cierre.

\paragraph{OE2: arquitectura e implementacion del prototipo.} Se obtuvo una arquitectura funcional basada en UI Toolkit, orquestacion de modos, gestores de escena y saneamiento del import mediante \texttt{ImportedDroneRuntimeBinder}. Ademas, el producto consolido una interfaz \textit{mobile-first} mejor fundamentada teoricamente que su adaptacion de escritorio y una capa visual basada en iconografia procedural, modos analiticos y presets de entorno. La separacion entre UI visible, codigo oculto y codigo no integrado mejora la trazabilidad del sistema y evita sobre-documentar capacidades que no forman parte de la version publicada.

\paragraph{OE3: evaluacion de usabilidad y carga de trabajo percibida.} Los instrumentos metodologicos quedaron definidos y alineados con el flujo final visible de la aplicacion. Sin embargo, la ejecucion de SUS, NASA-TLX Raw y \textit{Think-Aloud} debe realizarse sobre la build congelada posterior a la optimizacion final; por ello este objetivo no se presenta como empiricamente cerrado en esta version documental.

\paragraph{OE4: documentacion y transferencia.} Se dejo una base documental mas consistente entre propuesta, informe, manuales, arquitectura y app real. La normalizacion de conteos, escenas y estados funcionales reduce contradicciones y fortalece el cierre academico del proyecto.

\subsection{Limitaciones}

\begin{itemize}
    \item falta verificar en Unity el FBX runtime final con masters, instancias, fasteners, hotspots, filtros y helices;
    \item falta aplicar y validar las texturas/materiales finales antes del \textit{freeze};
    \item la evaluacion con usuarios aun no produce resultados finales reportables;
    \item el audio no forma parte del estado funcional verificado al momento de este cierre documental;
    \item varias herramientas existen en codigo pero no hacen parte de la UI final visible;
    \item la version de escritorio se mantiene como adaptacion funcional del diseno \textit{mobile-first}, sin un sistema de diseno especifico para escritorio plenamente desarrollado;
    \item la adopcion plena de una ruta nativa de LOD constituye una decision tecnica evaluada, no un cambio cerrado en la build documentada.
\end{itemize}

\subsection{Trabajo futuro}

\begin{enumerate}
    \item completar la verificacion del FBX runtime final en Unity y cerrar la importacion con reporte de jerarquia, instancias y grupos;
    \item rerun de auditorias de cobertura y jerarquia sobre la build final;
    \item ejecutar pruebas de usabilidad y consolidar el capitulo de resultados;
    \item integrar audio solo si entra antes del \textit{freeze} definitivo; en caso contrario, mantenerlo como trabajo futuro;
    \item evaluar la exposicion controlada de herramientas actualmente ocultas, como medicion, modos visuales adicionales y componentes de ensamblaje;
    \item calibrar y validar con datos experimentales el subsistema termico en iteraciones posteriores;
    \item decidir con datos de profiling si una ruta nativa de LOD aporta una mejora real al cierre del pipeline;
    \item incorporar soporte multi-idioma para ampliar el alcance del visor;
    \item incluir el cableado interno del dron en la taxonomia y la escena para completar la representacion del ensamblaje;
    \item automatizar el cuello de botella de optimizacion de modelos CAD, actualmente dependiente de intervencion manual en retopologia y saneamiento geometrico.
\end{enumerate}

\subsection{Recomendaciones}

\begin{enumerate}
    \item mantener una unica taxonomia documental para piezas, \textit{anchors} y \textit{renderers};
    \item documentar como finales solo las funciones visibles y verificables de la build congelada;
    \item cerrar la narrativa de resultados solo con datos reales, fechados y trazables;
    \item preservar el resto del codigo como base tecnica, pero distinguiendo siempre entre integrado, oculto y futuro.
\end{enumerate}
