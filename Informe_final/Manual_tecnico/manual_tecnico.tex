\section{Introduccion}
Este manual describe la arquitectura real de la build final documentada. El objetivo es registrar el funcionamiento del sistema visible, los modulos de soporte que siguen activos en runtime y la separacion entre codigo integrado, codigo oculto y codigo legado.

\section{Convenciones canonicas}

\subsection{Taxonomia de piezas}
\begin{itemize}
    \item \textbf{28 piezas canonicas:} entidades semanticas definidas en \texttt{x500v2\_parts\_data.json}.
    \item \textbf{30 anchors de escena:} 28 piezas + \texttt{x500v2\_fastener\_group} + \texttt{x500v2\_misc\_group}.
    \item \textbf{257 renderers/colliders:} granularidad tecnica del import final auditado.
    \item \textbf{257 assets generados:} archivos \texttt{.asset} en \texttt{Assets/Core/Data/X500V2Generated}.
\end{itemize}

\subsection{Taxonomia de scripts}
\begin{itemize}
    \item \textbf{95} scripts runtime propios en \texttt{Assets/Scripts} excluyendo \texttt{Editor/Tests}.
    \item \textbf{103} scripts bajo \texttt{Assets} excluyendo tests.
    \item \textbf{129} archivos \texttt{.cs} totales incluyendo editor y plugins.
\end{itemize}

\section{Escena y verdad funcional}

\begin{itemize}
    \item \textbf{Escena oficial:} \texttt{MainScene\_Final}
    \item \textbf{Layout UI:} \texttt{MainLayout.uxml}
    \item \textbf{Control principal UI:} \texttt{UIManager.cs}
    \item \textbf{Saneamiento runtime del import:} \texttt{ImportedDroneRuntimeBinder.cs}
    \item \textbf{Validacion funcional de referencia:} \texttt{VALIDACION\_FUNCIONAL\_FINA\_2026-04-09.md}
\end{itemize}

\subsection{Flujo visible}
\begin{center}
\texttt{Hero -> Explore -> seleccion -> bottom sheet -> Inspect / Analyze / Studio}
\end{center}

\section{Arquitectura runtime}

\subsection{Capas operativas}

\begin{enumerate}
    \item \textbf{Presentacion.} UI Toolkit (\texttt{MainLayout.uxml}, USS y controladores UI).
    \item \textbf{Orquestacion.} \texttt{UIManager}, \texttt{AppStateMachine}, \texttt{UIModeController}.
    \item \textbf{Servicios de escena.} seleccion, visibilidad, explode, clipping, view modes, entorno y subsistema termico.
    \item \textbf{Datos.} \texttt{DronePartData}, assets generados, grafo termico y catalogo canonico JSON.
\end{enumerate}

\subsection{Bootstrap}

El inicio del sistema combina dos mecanismos:

\begin{enumerate}
    \item \textbf{UIManager.EnsureManagers().} Garantiza la presencia de managers minimos para el visor.
    \item \textbf{ImportedDroneRuntimeBinder.} Repara jerarquia, reubica huerfanos, sincroniza anchors y reconstruye caches.
\end{enumerate}

\noindent El orden funcional esperado es:
\begin{enumerate}
    \item carga de \texttt{MainScene\_Final};
    \item inicializacion de \texttt{UIManager};
    \item garantia de managers base;
    \item ejecucion de \texttt{ImportedDroneRuntimeBinder};
    \item reconstruccion de caches de explode, visibilidad, catalogacion, clipping, view modes y thermal;
    \item activacion del flujo visible para el usuario.
\end{enumerate}

\section{Datos y normalizacion del modelo}

\subsection{Catalogo canonico}
El archivo \texttt{x500v2\_parts\_data.json} fija el inventario semantico del proyecto. Toda narrativa academica debe tomar este archivo como fuente para el numero canonico de piezas.

\subsection{Saneamiento del import}
\texttt{ImportedDroneRuntimeBinder} ejecuta las siguientes tareas:
\begin{itemize}
    \item resolver el root del dron importado;
    \item construir el mapa de anchors por \texttt{id};
    \item reubicar objetos huerfanos de nivel superior o anidados;
    \item crear o reutilizar grupos sinteticos para tornilleria y miscelaneos;
    \item reconstruir caches de managers dependientes;
    \item disparar el evento de runtime bound para regenerar hotspots o auditorias.
\end{itemize}

\section{Tooling de editor}

\section{Build y despliegue}

\begin{enumerate}
    \item abrir el proyecto en la version de Unity documentada por el repositorio;
    \item verificar la escena \texttt{MainScene\_Final};
    \item ejecutar las utilidades de configuracion necesarias;
    \item generar build WebGL;
    \item validar jerarquia, cobertura y funcionamiento visible antes de capturas o pruebas.
\end{enumerate}

\section{Limitaciones vigentes del cierre}

\begin{itemize}
    \item faltan optimizaciones CAD/import antes del freeze definitivo;
    \item los efectos de audio no hacen parte del estado final verificado;
    \item el capitulo de resultados depende de KPIs y pruebas de usabilidad posteriores al freeze;
    \item la herramienta de medicion no se expone hasta validar escala util del modelo.
\end{itemize}
